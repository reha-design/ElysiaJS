Elysia: 자바스크립트 웹 프레임워크 최적화 

**저자:** Chayapatr Archiwaranguprok, Kongkeit Khunpanitchot, Phoomparin Mano, Manachai Toahchoodee 

---

초록 (ABSTRACT) 자바스크립트와 그 백엔드 웹 프레임워크가 주류로 채택되고 오늘날 업계에서 중추적인 역할을 함에 따라, 런타임 성능, 개발자 경험, 타입 안전성의 과제는 확장성과 유지보수성을 저해하는 잠재적 요인이 되고 있습니다. 본 논문은 정적 분석 및 동적 코드 생성 기법을 통한 JIT(Just-In-Time) 컴파일과 자바스크립트의 인터프리터 특성을 활용하여 잠재적으로 비용이 많이 드는 인체공학적 기능의 성능을 최적화함으로써 성능과 편의성 사이의 균형을 제공하는 자바스크립트 웹 프레임워크 전략을 제안합니다. 또한 TypeScript의 타입 시스템을 채택하여 향상된 개발자 경험을 제공합니다. 개발자 경험을 위한 최적화 전략 및 설계의 참조 구현은 Elysia 자바스크립트 웹 프레임워크의 구현을 통해 예시되며, 여기서 성능은 최적화되지 않은 코드와 평가되고 다른 실제 백엔드 웹 프레임워크와 벤치마킹됩니다.

1. 서론 (INTRODUCTION) 지난 10년 동안 자바스크립트는 백엔드 웹 개발을 위한 프로그래밍 언어로 주류로 채택되었습니다. 자바스크립트는 브라우저에서 사용하도록 설계되었지만, Node.js 및 Express와 같은 자바스크립트 런타임 및 프레임워크의 출현으로 자바스크립트가 널리 사용될 수 있게 되었습니다. 2023년 Stack Overflow 개발자 설문조사에 따르면, 자바스크립트는 87,585명의 응답자 중 63.61%가 사용하여 11년 연속 가장 많이 사용되는 프로그래밍 언어의 자리를 지키고 있습니다. 자바스크립트로 개발된 백엔드 프레임워크는 "웹 기술" 범주에서 가장 인기 있는 프레임워크 중 하나이며, Node.js 런타임이 71,802명의 응답자 중 42.65%로 1위를 차지했고, Express 프레임워크가 19.28%로 4위를 차지했습니다. 이는 C#의 ASP.NET Core(16.57%), Python의 Flask(12.16%), Java의 Spring Boot(11.95%), PHP의 Laravel(7.58%)과 같은 다른 언어의 프레임워크와 비교할 때 가장 인기 있는 수치입니다. 관찰된 통계는 그림 1에 표시된 것처럼 이전 연도의 설문 조사와 유사합니다.

그러나 자바스크립트 웹 프레임워크의 성능을 다른 언어와 비교할 때 기존 프레임워크의 성능 오버헤드는 상당합니다. TechEmpower의 21라운드 성능 벤치마크에 따르면, Express는 종합 테스트 점수에서 137개 중 113위를 차지한 반면, Nest는 85위, Fastify는 76위를 차지했습니다. 이는 자바스크립트에서 백엔드 웹 프레임워크를 최적화하는 데 있어 과제를 안겨줍니다.
그럼에도 불구하고 Donald Knuth가 "섣부른 최적화는 모든 악의 근원이다"라고 썼듯이, 성능 향상이 장기적으로 개발자 생산성과 코드 유지보수성을 저하시키는 비용을 초래해서는 안 됩니다. 따라서 최적화보다 프레임워크 설계의 더 많은 측면을 함께 고려해야 합니다. 목표는 성능, 개발자 경험, 타입 안전성의 핵심 원칙을 조화시키는 것이어야 합니다.
균형을 달성하기 위해 본 논문에서는 위에서 언급한 과제를 해결하는 최적화 전략 및 개발자 경험 중심 설계의 실용적인 참조 구현 역할을 하는 TypeScript 웹 프레임워크인 Elysia 프레임워크를 소개합니다. 또한 이 논문에서는 최적화되지 않은 코드와 비교하여 최적화 성능을 평가하고 다른 실제 백엔드 웹 프레임워크와 비교합니다.

2. 관련 연구 (RELATED WORKS) 

* 
2.1 TypeScript : TypeScript는 Microsoft에서 자바스크립트의 엄격한 구문적 상위 집합으로 개발한 오픈 소스 프로그래밍 언어입니다. TypeScript의 타입 시스템을 통해 언어 컴파일러는 컴파일 시간에 많은 자바스크립트 버그를 감지할 수 있습니다. TypeScript의 타입 시스템은 개발자에게 타입 검사 및 자동 완성을 제공하여 개발자 경험과 생산성을 향상시킵니다. TypeScript는 또한 타입 주석에 명확한 인터페이스를 제공하여 데이터 타입에 대한 예상을 전달함으로써 개발자 간의 오해를 줄여 코드 유지 관리성을 향상시킵니다. 이는 정의되지 않은 값이나 타입 강제 변환과 같은 미묘한 버그를 유발하지 않고 코드를 리팩토링하거나 확장하기 쉽게 만듭니다. TypeScript는 자바스크립트의 상위 집합이므로 자바스크립트로 컴파일하여 프론트엔드 및 백엔드 애플리케이션에서 활용할 수 있습니다.


* 
2.2 자바스크립트 런타임 : 자바스크립트 런타임은 자바스크립트 엔진을 파일 시스템 및 HTTP 서버와 같은 구성 요소와 통합하여 백엔드 웹 개발 환경을 만듭니다. 2009년에 도입된 자바스크립트 런타임인 Node.js는 서버 측에서 자바스크립트 채택을 촉진하는 데 중추적인 역할을 했습니다. Node.js는 V8 자바스크립트 엔진을 활용하여 이벤트 루프와 비동기 I/O를 위한 API를 제공합니다. Bun, Deno, Netlify Functions, Vercel Functions와 같은 최신 자바스크립트 런타임은 런타임 성능, 개발자 경험 및 사용 편의성을 향상시키기 위해 개발되었습니다. Elysia.js는 주로 Zig 언어로 작성된 JavaScriptCore 엔진으로 구축되어 2022년에 도입된 런타임인 Bun에 최적화되어 있습니다.


* 
2.3 자바스크립트 웹 프레임워크의 단계 : 자바스크립트 백엔드 웹 프레임워크는 HTTP 요청 처리 및 데이터베이스 관리와 같이 웹 애플리케이션 구축을 위해 미리 정의된 함수와 유틸리티가 있는 구조화된 환경을 제공합니다. 이를 통해 개발자는 모범 사례를 따르고, 시간을 절약하며, 코드 일관성을 유지할 수 있습니다. 다양한 설계와 장단점을 가진 여러 웹 프레임워크가 자바스크립트용으로 만들어졌습니다. 논리적으로 백엔드 웹 프레임워크는 초기화와 요청 처리라는 최소 두 단계를 거칩니다. 초기화 시 프레임워크는 HTTP 메서드와 경로 일치자를 경로 핸들러와 연결하여 라우팅 테이블을 구성하고, 요청을 처리할 미들웨어 함수를 등록하며, 요청을 수락하기 위해 HTTP 서버를 시작합니다. 그런 다음 각 요청에 대해 프레임워크는 요청을 객체(예: 헤더, 본문, 쿠키)로 파싱하고, 일치하는 라우트를 확인하며, 파싱된 데이터를 각 미들웨어 계층으로 전달한 후 요청을 클라이언트로 다시 보냅니다.


* 
2.4 엔드투엔드 타입 안전성 : 엔드투엔드 타입 안전성은 프론트엔드, 백엔드, 데이터베이스 등 전체 애플리케이션 스택에서 타입 안전성을 보장하는 것을 목표로 하는 방식입니다. 이는 모든 단계에서 런타임 오류를 줄이고 애플리케이션 전반에 걸쳐 일관된 타입 추론 및 타입 검사를 제공하여 개발자 생산성을 향상시키는 것을 목표로 합니다. RESTful API는 클라이언트와 서버 간 통신에 자주 사용되지만, 기본적으로 타입 안전성을 강제하지 않는다는 단점이 있습니다. 또한 클라이언트, 서버, 데이터베이스 스키마의 타입이 개별적으로 정의되는 경우가 많아 불일치와 오류가 발생합니다. 예를 들어 백엔드의 타입이 변경되면 클라이언트는 런타임까지 변경 사항을 정적으로 감지할 수 없어 런타임 오류가 발생합니다. 엔드투엔드 타입 안전성을 달성하기 위해 SOAP, Apache Thrift, JSON-RPC, gRPC, GraphQL과 같은 다양한 기술이 개발되었으며, API 인터페이스를 설명하기 위해 스키마를 한 번 정의하면 코드 생성 도구가 타입이 지정된 클라이언트 및 서버 스텁을 생성합니다. 그러나 코드 생성을 구성하는 데 필요한 도구 및 빌드 단계는 종종 추가적인 복잡성을 수반합니다.


* 
2.5 자바스크립트 최적화 도구 : Google Closure Compiler 및 Prepack을 포함하여 자바스크립트 코드의 성능을 향상시키기 위해 자바스크립트 생태계에서 여러 도구가 개발되고 있습니다. 이러한 도구는 자바스크립트 코드를 분석하고 컴파일 시간 평가, 죽은 코드 제거, 함수 인라인화와 같은 최적화를 적용합니다. 그러나 이 접근 방식은 애플리케이션의 빌드 프로세스에 추가 구성이 필요하며, 애플리케이션 개발자는 도구와 최적화 플래그가 소스 코드에서 올바르게 작동하는지 확인하기 위해 최적화를 구성하고 테스트해야 합니다. 이와 대조적으로 본 논문에서는 추가 빌드 단계가 필요 없는 JIT(Just-In-Time) 컴파일을 사용하는 런타임 기반 방법을 제시합니다. 웹 프레임워크가 함수를 안전하게 최적화하는 데 필요한 도메인 지식을 보유하고 있기 때문에, 최종 사용자의 추가 구성 없이 표적화된 방식으로 최적화를 적용할 수 있는 수단을 제공합니다.



3. Elysia 프레임워크 설계 (DESIGN OF ELYSIA FRAMEWORK) 성능과 사용 편의성 사이에서 절충하는 대신 JIT 컴파일의 동작을 활용하여 서로를 희생하지 않고 리소스 소모가 큰 인체공학적 기능의 성능을 최적화할 수 있습니다. Elysia는 요청 파싱, 문서 생성, 엔드투엔드 타입 안전성, 스키마를 통한 엄격한 검증과 같은 인체공학적 기능을 구축하고, JIT 컴파일을 통한 정적 분석 및 동적 코드 생성과 같은 최적화 전략을 적용하여 성능 오버헤드를 제거함으로써 성능과 편의성에 대한 균형 잡힌 초점을 제공하는 것을 목표로 합니다. 성능 향상을 위해 개발 중인 Bun 및 Deno와 같은 여러 최신 런타임이 있으므로 런타임 전반에 걸친 호환성을 제공할 뿐만 아니라 네트워크 지연 시간을 줄이기 위해 엣지 환경에 대한 지원을 제공합니다. Elysia는 Bun, Deno, Netlify Edge, Vercel Edge 런타임과 같은 여러 최신 웹 런타임에서 구현되는 표준인 WinterCG 사양을 구현하여 런타임 전반에 걸쳐 HTTP 요청 핸들러에 대한 단일 통합 인터페이스를 제공합니다.

* 
3.1 사전 컴파일 적용 : Elysia는 프레임워크 초기화 단계에서 자바스크립트 엔진의 JIT 컴파일러를 활용하기 위해 사전 컴파일(AOT)을 사용합니다. Elysia 애플리케이션이 초기화될 때 프레임워크는 라이프사이클 훅을 반복하여 프레임워크가 HTTP 요청을 수신할 때 호출되는 개별 경로에 대한 요청 처리를 최적화하는 핸들러 함수를 동적으로 구성합니다. 이는 라우트 함수에서 toString 메서드를 사용하여 라이프사이클 훅 함수의 소스 코드를 내부 분석(introspection)함으로써 수행되며, 이를 통해 Elysia는 어떤 컨텍스트 인수가 참조되는지 결정할 수 있습니다. 라이프사이클 훅을 통해 사용자는 요청 파이프라인의 이벤트에 연결하여 사용자 지정 본문 파서 및 인증 가드와 같은 추가 동작을 구현할 수 있습니다. 그런 다음 Elysia는 파라메트릭 라우트를 저장하기 위한 기수 트리(radix tree)를 구성하고 동적 코드 생성을 사용하여 switch 문을 활용해 정적 라우트와 일치하는 함수를 컴파일합니다.


* 
3.2 내부 분석 및 코드 생성 : 정적 분석은 실행 없이 소스 코드를 분석하는 데 사용되는 기술입니다. 자바스크립트는 인터프리터 언어이므로, 코드의 문자열 표현을 반환하는 Function 인스턴스의 toString 메서드 사용을 통해 런타임 시 함수의 소스 코드를 정적으로 분석할 수 있습니다. 또한 Function 생성자를 사용하면 자바스크립트 엔진이 소스 코드의 문자열 표현을 기반으로 런타임 시 함수의 JIT 컴파일을 수행할 수 있습니다.


* 
3.3 요청 파싱 오버헤드 : HTTP 프레임워크는 라우트 핸들러에서 사용하기 위해 HTTP 요청을 파싱해야 합니다. 기존 프레임워크는 요청의 모든 부분을 적극적으로 파싱하고 콘텐츠 유형을 동적으로 감지하는 두 가지 상당한 성능 불이익을 초래합니다. 프레임워크는 라우트가 어떤 부분을 사용할지 미리 알지 못하기 때문에 모든 것을 미리 파싱하며 , 매 요청마다 Content-Type 헤더를 감지해 파싱을 진행하여 성능을 떨어뜨립니다.


* 
3.4 조건부 요청 파싱 : 이 오버헤드를 없애기 위해 정적 분석 기법을 사용하여 라우트 핸들러가 실제로 참조하는 요청 부분만을 식별합니다. 그런 다음 Elysia는 사용 패턴을 기반으로 참조된 요청 부분만 조건부로 파싱하는 함수를 구성합니다. 또한 개발자가 강력한 스키마를 미리 지정하도록 하여 자동 검증 및 자동 완성을 제공합니다.


* 
3.5 인자 참조 감지 : 함수가 어떤 인자를 참조하는지 감지하기 위해 점 표기법, 대괄호 표기법, 객체 구조 분해 및 별칭 참조 여부를 정규식을 통해 패턴 매칭합니다.


* 
3.6 속성 할당 인라인화 : 라이프사이클 훅에서 컨텍스트 객체를 병합할 때 객체 확산 연산자를 사용하면 오버헤드가 발생합니다. 프레임워크는 컨텍스트 객체의 형태를 내부 분석하여 객체 내에 인라인 키-값 매핑을 동적으로 생성하여 오버헤드를 제거합니다.


* 
3.7 정적 라우트 매칭 인라인화 : 정적 라우트를 해시맵에서 조회하는 속도는 엔진마다 다릅니다. Elysia는 AOT 컴파일 단계에서 정적 라우트를 해당 핸들러 함수에 매핑하는 switch 문을 문자열 리터럴로 생성하여 JIT 컴파일 속도를 높입니다.


* 
3.8 조건부 Await 표현식 : 라이프사이클 훅이 수없이 실행될 때 불필요한 await 구문은 마이크로태스크 큐에 연속 콜백을 추가해 병목을 일으킵니다. Elysia는 함수가 AsyncFunction인지 검사하여 동기 함수일 경우 await 키워드를 조건부로 제거해 최적화합니다.


* 
3.9 스키마를 통한 요청 파싱 : 개발자가 라우트에 스키마 타입을 명시하면 프레임워크는 이 타입을 AOT 단계에서 분석하여 필요한 파서만 주입하므로, 런타임에 콘텐츠 유형을 감지하는 비용을 없앱니다.


* 
3.10 스키마를 위한 커스텀 타입 : @sinclair/typebox 라이브러리를 활용해 스키마를 정의합니다. Elysia가 제공하는 Numeric, URLEncoded, Files 등의 커스텀 타입은 컴파일 타임 추론과 런타임 파싱을 동시에 제공하며, 자동화된 Swagger 문서 생성에도 쓰입니다.


* 
3.11 타입 추론을 위한 API 설계 : Elysia의 state, decorate, derive 등의 메서드는 TypeScript의 제네릭 타입을 활용하여 사용자가 정의한 데이터의 타입을 훅이나 라우트 전체로 안전하게 전파합니다.


* 
3.12 엔드투엔드 타입 안전성 : 별도의 코드 생성 도구 없이, 자바스크립트의 Proxy 객체와 TypeScript의 타입 수준 변환(Type-level transformations)을 결합한 Eden Treaty 클라이언트를 제공하여 서버 측 스키마를 클라이언트에서 곧바로 추론하고 동기화할 수 있게 합니다.



4. 정적 분석 및 동적 코드 생성의 영향 (IMPACT ON STATIC ANALYSIS AND DYNAMIC CODE GENERATION) 최적화가 특정 엔진에 국한되지 않음을 입증하기 위해 Bun, Deno, Node.js 세 가지 런타임 환경에서 벤치마크를 실행했습니다. 조건부 요청 파싱, 속성 할당 인라인화, 정적 라우트 매칭 인라인화, 조건부 await 제거 등의 최적화 기법은 모든 환경에서 유의미하고 큰 성능 향상을 보였습니다. 다만 Node.js의 경우 아직 WinterCG fetch 사양을 네이티브로 지원하지 않아 요청 변환 오버헤드로 인해 다른 런타임보다 요청 파싱이 느린 결과를 보였습니다.
