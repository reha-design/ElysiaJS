## 초록

JavaScript와 백엔드 웹 프레임워크가 주류로 자리 잡으면서, 런타임 성능·개발자 경험·타입 안전성이 확장성과 유지보수성의 걸림돌이 되고 있다. 본 논문은 JavaScript의 인터프리터 특성과 JIT(Just-In-Time) 컴파일을 활용한 정적 분석 및 동적 코드 생성 기법을 통해 성능과 편의성의 균형을 달성하는 전략을 제시한다. 이 전략의 참조 구현체가 바로 Elysia 프레임워크이며, 최적화 전후 성능 비교 및 타 프레임워크 벤치마크 결과도 함께 평가한다.

---

## 1. 서론

지난 10년간 JavaScript는 백엔드 웹 개발 언어로 주류에 진입했다. Stack Overflow 개발자 설문 2023에 따르면 JavaScript는 11년 연속 가장 많이 사용되는 언어(63.61%)이며, Node.js는 웹 기술 분야 1위(42.65%), Express는 4위(19.28%)를 기록하고 있다.

그러나 JavaScript 프레임워크의 성능은 타 언어 대비 현저히 낮다. TechEmpower 라운드 21 벤치마크에서 Express는 137개 중 113위, Nest는 85위, Fastify는 76위에 불과하다.

이에 본 논문은 Donald Knuth의 "섣부른 최적화는 만악의 근원"이라는 경고를 염두에 두고, **성능·개발자 경험·타입 안전성**을 동시에 추구하는 Elysia 프레임워크를 소개한다.

## 2. 관련 연구

### 2.1 TypeScript
TypeScript는 Microsoft가 개발한 JavaScript의 상위 집합 언어로, 컴파일 타임에 많은 버그를 탐지하고 타입 검사 및 자동 완성을 통해 개발 생산성과 코드 유지보수성을 향상시킨다.

### 2.2 JavaScript 런타임
Node.js(2009)는 V8 엔진 기반으로 서버 측 JavaScript 채택을 이끌었다. Bun, Deno 등 최신 런타임은 성능과 개발 편의성 향상을 목표로 개발되고 있으며, Elysia는 특히 2022년 등장한 Zig 언어 기반의 Bun 런타임에 최적화되어 있다.

### 2.3 웹 프레임워크의 단계
백엔드 웹 프레임워크는 최소 두 단계로 구성된다.
- **초기화 단계**: 라우팅 테이블 구성, 미들웨어 등록, HTTP 서버 시작
- **요청 처리 단계**: 요청 파싱, 라우트 매칭, 미들웨어 실행, 응답 반환

### 2.4 엔드투엔드 타입 안전성
RESTful API는 기본적으로 타입 안전성을 강제하지 않아 클라이언트·서버·DB 간 타입 불일치 문제가 발생한다. SOAP, gRPC, GraphQL 등이 해결책으로 등장했으나 코드 생성 설정의 복잡성이 단점이다.

### 2.5 JavaScript 최적화 도구
Google Closure Compiler, Prepack 등의 빌드 타임 최적화 도구와 달리, Elysia는 **런타임 기반 JIT 컴파일** 방식을 채택하여 추가 빌드 설정 없이 최적화를 적용한다.

## 3. Elysia 프레임워크 설계

성능과 편의성 사이의 트레이드오프 없이, JIT 컴파일의 특성을 활용해 고비용 편의 기능의 성능 오버헤드를 제거하는 것이 핵심 설계 철학이다.

Elysia는 WinterCG 표준을 구현하여 Bun, Deno, Netlify Edge, Vercel Edge 등 다양한 런타임에서 단일 인터페이스로 동작한다.

### 3.1 AOT(Ahead-of-Time) 컴파일 적용
초기화 단계에서 라이프사이클 훅을 순회하며 각 라우트에 최적화된 핸들러 함수를 동적으로 생성한다. 함수의 `toString()` 메서드로 소스 코드를 분석해 어떤 컨텍스트 인자가 참조되는지 파악한다.

### 3.2 인트로스펙션과 코드 생성
JavaScript의 인터프리터 특성을 이용해 `Function.toString()`으로 소스 코드를 런타임에 정적 분석하고, `Function` 생성자로 최적화된 코드를 JIT 컴파일한다.

### 3.3 요청 파싱 오버헤드
기존 프레임워크는 두 가지 성능 문제를 안고 있다.
1. **과도한 파싱**: 라우트가 사용하지 않는 요청 부분까지 항상 파싱
2. **런타임 Content-Type 감지**: 매 요청마다 Content-Type을 동적으로 확인

### 3.4 조건부 요청 파싱
정적 분석으로 라우트 핸들러가 실제로 참조하는 요청 부분(body, query, headers, cookies 등)만 파싱하도록 핸들러 함수를 동적으로 구성한다.

### 3.5 인자 참조 감지
함수가 특정 인자를 참조하는지 다음 네 가지 방식으로 판별한다.
1. 인자 없음 여부
2. 객체 구조 분해(destructuring) 사용 여부
3. 점 표기법 또는 괄호 표기법 사용 여부
4. 별칭(alias) 변수를 통한 참조 여부

### 3.6 프로퍼티 할당 인라이닝
`Object.spread`나 `Object.assign` 대신 초기화 시점에 컨텍스트 객체의 형태를 분석하여 프로퍼티 할당을 인라인으로 생성함으로써 오버헤드를 제거한다.

### 3.7 정적 라우트 매칭 인라이닝
정적 라우트는 해시맵에, 동적(파라메트릭) 라우트는 Radix Tree에 저장한다. 일부 JavaScript 엔진에서는 해시맵보다 `switch` 문이 더 빠르므로, 런타임 환경에 따라 동적으로 최적화된 `switch` 문을 AOT 컴파일 단계에서 생성한다.

### 3.8 조건부 Await 표현식
라이프사이클 훅이 동기 함수인지 비동기 함수인지를 초기화 시점에 파악하여, 동기 함수에는 `await`를 붙이지 않도록 한다. 불필요한 `await`는 마이크로태스크 큐 오버헤드를 유발하기 때문이다.

### 3.9 스키마를 통한 요청 파싱
라우트에 스키마를 지정하면 Content-Type 감지 오버헤드 없이 AOT 컴파일 단계에서 적절한 파서 코드를 미리 생성한다.

### 3.10 스키마를 위한 커스텀 타입
`@sinclair/typebox` 라이브러리 기반으로 `Numeric`, `URLEncoded`, `File/Files`, `Cookie` 등의 커스텀 타입을 제공하며, 런타임 파싱과 컴파일 타임 타입 추론을 동시에 지원한다. 스키마는 OpenAPI/Swagger 문서 자동 생성에도 활용된다.

### 3.11 타입 추론을 위한 API 설계
`state`, `decorate`, `derive` 등의 메서드는 TypeScript 제네릭 타입을 활용해 복잡한 중첩 라우트, 플러그인, 에러 상태 등에 대한 완전한 타입 추론을 제공한다.

### 3.12 엔드투엔드 타입 안전성
별도 코드 생성 없이 서버의 TypeScript 타입 정의를 클라이언트에서 직접 임포트하는 방식으로 엔드투엔드 타입 안전성을 달성한다. **Eden Treaty** 클라이언트는 JavaScript Proxy 객체와 TypeScript 타입 변환을 조합하여 `app.users[12].post()` 같은 타입 안전한 API 호출을 지원한다.

## 5. 논의 및 향후 연구

2023년 9월 기준 Elysia는 GitHub 스타 약 5,000개, npm 주간 다운로드 약 10,000회를 기록하며 상당한 관심을 받고 있다.

향후 연구로는 Rust의 절차적 매크로(procedural macro)와 같이 다른 언어에서도 정적 분석 및 동적 코드 생성 기법을 적용하는 방향을 탐색할 계획이다.
