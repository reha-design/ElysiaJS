# Redis 설계 원칙 및 캐싱 가이드

Redis를 백엔드 캐시 시스템으로 사용할 때 성능을 극대화하고 메모리 효율을 높이기 위한 지침입니다.

---

## 1. 명확한 키 네이밍 규칙 (Key Naming)
Redis는 키-값 저장소이므로 키 이름만 보고도 데이터를 알 수 있어야 합니다.

- **계층 구조:** 콜론(:)을 사용하여 계층을 구분합니다. (예: posts:all, user:100:profile)
- **가독성:** 너무 길지 않으면서도 의미 있는 이름을 사용합니다.
- **접두어 활용:** 프로젝트나 환경에 따라 접두어를 붙여 충돌을 방지합니다.

## 2. 적절한 데이터 타입 선택
데이터의 특성에 맞는 타입을 사용하여 메모리를 절약합니다.

- **String:** 단순 값, JSON 문자열, 세션 정보 (가장 기본)
- **Hash:** 객체 형태의 데이터 (필드별 수정이 필요할 때 유리)
- **List:** 스택/큐 기능, 최근 활동 로그
- **Set:** 중복 없는 고유값 모음 (좋아요 한 유저 ID 등)
- **Sorted Set:** 점수 기반의 랭킹 시스템

## 3. TTL (Time To Live) 설정 필수
메모리 부족을 방지하고 데이터의 신선도를 유지하기 위해 모든 캐시에는 만료 시간을 설정합니다.

- **휘발성 데이터:** 세션이나 일시적인 토큰은 짧은 TTL을 부여합니다.
- **조회성 데이터:** 데이터베이스 부하가 큰 정보는 적절한 TTL(예: 1분~1시간)을 주어 DB 호출을 최소화합니다.

## 4. 캐싱 전략 (Caching Strategies)
- **Look-aside (Lazy Loading):** 이번 프로젝트에서 사용할 방식입니다. 
    1. 애플리케이션이 캐시 확인 
    2. 있으면 반환(Cache Hit) 
    3. 없으면 DB 조회 후 캐시에 저장(Cache Miss)
- **Write-through:** 데이터 작성 시 DB와 캐시에 동시에 쓰는 방식입니다. (데이터 일치성 보장)

## 5. 캐시 무효화 (Cache Invalidation)
가장 어려운 부분 중 하나로, 원본 데이터(DB)가 바뀔 때 캐시를 어떻게 처리할지 결정해야 합니다.

- **전략:** 데이터 수정 시 관련 캐시를 즉시 삭제(DEL)하거나, TTL이 만료될 때까지 기다립니다.
- **주의:** 캐시를 삭제하지 않으면 사용자가 한동안 예전 데이터를 보게 됩니다.

## 6. 메모리 관리와 빅 키(Big Keys) 주의
Redis는 인-메모리 데이터베이스이므로 메모리 관리가 생명입니다.

- **큰 데이터 지양:** 하나의 키에 너무 큰 데이터(수십 MB 이상)를 넣지 않습니다.
- **KEYS 명령어 금지:** 서버가 멈출 수 있으므로 운영 환경에서는 SCAN 명령어를 사용합니다.

## 7. 가용성과 백업 (RDB vs AOF)
데이터 보존 필요성에 따라 백업 방식을 선택합니다.

- **RDB (Snapshot):** 특정 시점의 스냅샷을 저장 (재시작 빠름)
- **AOF (Append Only File):** 모든 쓰기 명령을 기록 (데이터 유실 최소화)
