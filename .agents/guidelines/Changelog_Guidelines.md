# Changelog 업데이트 가이드라인

이 문서는 프로젝트에 새로운 기능이 추가되거나 변경 사항이 발생했을 때, `CHANGELOG.md` 파일을 업데이트하는 표준 절차를 정의합니다.

## 1. 업데이트 시점
*   **기능 완성 직후**: 새로운 API 엔드포인트, 비즈니스 로직, 또는 UI 컴포넌트 개발이 완료되었을 때.
*   **중요 설정 변경**: 데이터베이스 스키마, 환경 변수, 또는 주요 라이브러리 설정이 변경되었을 때.
*   **이슈 해결**: 버그 수정이나 트러블슈팅 완료 후 해당 내용을 기록할 때.

## 2. 기록 양식 및 분류
항상 최신 내용이 문서의 **최상단**에 위치하도록 하며, 다음 카테고리를 준수합니다.

### [날짜] - [버전 번호]
*   **🚀 추가 기능 (Added)**: 새롭게 추가된 기능 명시.
*   **🛠 인프라 및 환경 (Infra & Env)**: 기술적 환경 변화, 라이브러리 추가 등.
*   **🐞 해결된 이슈 (Fixed)**: 수정된 버그 또는 해결된 기술적 문제.
*   **⚠️ 주의 사항 (Caution)**: 하위 호환성 문제나 운영 시 주의할 점 (선택 사항).

## 3. 작성 요건 (상세)
1.  **구체성**: 단순히 "기능 추가"라고 적지 않고, "IORedis를 활용한 Posts 조회 캐싱 구현"과 같이 구체적으로 작성합니다.
2.  **연결성**: 필요한 경우 관련 문서(예: `docs/troubleshooting/`)의 링크를 포함합니다.
3.  **일관성**: 기존의 톤앤매너(현재 프로젝트는 한글 기반)를 유지하며 정중하고 명확한 문장을 사용합니다.

## 4. 에이전트 의무 사항
*   에이전트는 모든 주요 작업의 마지막 단계에서 반드시 **`CHANGELOG.md` 업데이트 여부를 스스로 점검**해야 합니다.
*   업데이트 후에는 Git 커밋에 해당 내용을 포함시켜 원격 저장소와 항상 동기화합니다.
