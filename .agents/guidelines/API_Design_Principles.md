# RESTful API 설계 원칙 가이드

누구나 이해하기 쉽고, 유지보수가 용이하며, 클라이언트가 사용하기 편리한 API를 만들기 위한 핵심 설계 원칙입니다.

---

## 1. 리소스 중심의 URI (Resources)
API는 '동작'이 아닌 **'데이터(리소스)'**를 중심으로 경로를 설계해야 합니다.

- **명사 사용:** `/getPosts` (X) -> `/posts` (O)
- **복수형 권장:** 리소스가 컬렉션임을 명확히 하기 위해 복수형 사용을 권장합니다.
- **계층 관계 표현:** `/posts/1/comments` (1번 게시글의 댓글 목록)

## 2. HTTP 메서드의 올바른 사용
상태를 변경하는 행위는 URI가 아닌 **HTTP 메서드**를 통해 정의합니다.

- **GET:** 리소스 조회 (데이터 변경 없음)
- **POST:** 새 리소스 생성
- **PUT / PATCH:** 기존 리소스 수정 (전체 수정 / 부분 수정)
- **DELETE:** 리소스 삭제

## 3. 적절한 상태 코드 (Status Codes)
응답의 결과를 숫자로 명확히 전달하여 클라이언트가 상태를 즉시 파악하게 합니다.

- **200 (OK):** 요청 성공
- **201 (Created):** 생성 성공 (POST 요청 결과)
- **400 (Bad Request):** 클라이언트 요청 오류 (잘못된 파라미터 등)
- **401 (Unauthorized):** 인증 실패
- **404 (Not Found):** 요청한 리소스가 없음
- **500 (Internal Server Error):** 서버 내부 오류

## 4. 무상태성 (Statelessness)
각 요청은 독립적이어야 하며, 서버는 클라이언트의 이전 상태를 저장하지 않습니다.

- 모든 요청은 처리에 필요한 **모든 정보(인증 토큰 등)**를 포함해야 합니다.
- 서버의 확장성(Horizontal Scaling)을 높여줍니다.

## 5. 일관성 (Consistency)
데이터 형식과 네이밍 규칙은 전체 API에서 통일되어야 합니다.

- **데이터 형식:** 대부분 JSON 형식을 표준으로 사용합니다.
- **네이밍 컨벤션:** 프로젝트 전체에서 `camelCase` 또는 `snake_case` 중 하나를 선택하여 일관되게 사용합니다.

## 6. 버저닝 (Versioning)
API 스펙 변경으로 인한 기존 클라이언트의 중단을 막기 위해 버전을 명시합니다.

- **URI 방식:** `/v1/posts`, `/v2/posts` (가장 직관적이고 널리 쓰임)
- **Header 방식:** `Accept: application/vnd.myapi.v1+json`

## 7. 필터링, 정렬, 페이징
대량의 데이터를 처리할 때는 쿼리 파라미터를 사용합니다.

- **검색:** `/posts?q=elysia`
- **정렬:** `/posts?sort=-createdAt`
- **페이징:** `/posts?page=1&limit=10`

---

## 💡 보안 및 최적화
- **HTTPS 사용:** 데이터 전송 시 암호화는 필수입니다.
- **속도 제한 (Rate Limiting):** 악의적인 반복 요청으로부터 서버를 보호합니다.
- **적절한 캐싱:** `ETag` 등을 활용하여 불필요한 데이터 전송을 줄입니다.
