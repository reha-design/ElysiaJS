# 관계형 데이터베이스(RDB) 설계 원칙 가이드

안정적이고 성능이 뛰어난 데이터베이스를 구축하기 위해서는 단순한 '데이터 저장' 이상의 설계 고민이 필요합니다. 아래는 RDB 설계 시 반드시 지켜야 할 핵심 원칙입니다.

---

## 1. 정규화 (Normalization)
데이터 중복을 최소화하고 데이터 무결성을 유지하기 위한 과정입니다.

- **제1정규형 (1NF):** 모든 원자값(Atomic value)은 하나임(한 칸에 하나의 데이터만 입력).
- **제2정규형 (2NF):** 부분 함수적 종속성 제거(기본키의 일부에만 종속된 열이 없어야 함).
- **제3정규형 (3NF):** 이행적 함수적 종속성 제거(기본키가 아닌 열끼리 종속되면 안 됨).
- **효과:** 데이터 수정 시 이상 현상(삽입/삭제/수정 이상)을 방지합니다.

## 2. 고유 식별자 설정 (Primary Keys)
모든 테이블은 각 행을 유일하게 식별할 수 있는 **기본키(Primary Key)**를 가져야 합니다.

- **불변성:** 한 번 부여된 기본키는 절대 변경되어서는 안 됩니다.
- **최소성:** 식별에 꼭 필요한 최소한의 열로 구성합니다.
- **팁:** 비즈니스 로직(이메일, 주민번호 등)을 가진 값보다는 자동 증가하는 ID(Auto Increment)를 PK로 사용하는 것이 유지보수에 유리합니다.

## 3. 참조 무결성 (Referential Integrity)
테이블 간의 유효한 관계를 유지하기 위한 원칙입니다.

- **외래키(Foreign Key):** 관계를 맺는 테이블은 반드시 외래키 제약 조건을 설정합니다.
- **고아 데이터 방지:** 부모 테이블에서 행이 삭제될 때 자식 데이터가 남지 않도록 `ON DELETE CASCADE` 또는 `SET NULL` 등의 규칙을 명확히 정의합니다.

## 4. 데이터 타입의 정확성 (Domain Integrity)
각 열에 맞는 최적의 데이터 타입을 선택합니다.

- **용량 최적화:** 너무 큰 데이터 타입은 서버와 메모리 자원을 낭비합니다.
- **의미 전달:** 날짜 데이터는 `DATETIME` 또는 `TIMESTAMP`를, 가변 문자열은 `VARCHAR`를 적절히 사용합니다.
- **값의 범위:** 성별, 상태(Status) 등 값이 한정된 경우 `ENUM`이나 공통 코드 테이블을 활용합니다.

## 5. ACID 원칙 준수
트랜잭션(Transaction) 처리에 있어 데이터의 신뢰성을 보장하는 핵심 성질입니다.

- **Atomicity (원자성):** 전체가 다 성공하거나, 아니면 하나도 실행되지 않아야 함 (All or Nothing).
- **Consistency (일관성):** 트랜잭션 전후의 DB 상태가 항상 유효한 상태여야 함.
- **Isolation (격리성):** 동시에 실행되는 여러 트랜잭션이 서로 간섭하지 않아야 함.
- **Durability (지속성):** 성공한 트랜잭션의 결과는 시스템 장애가 나더라도 영구히 보관되어야 함.

---

## 💡 성능과 설계 사이의 트레이드 오프 (Denormalization)
실무에서는 정규화를 너무 완벽하게 할 경우, 수많은 **JOIN**으로 인해 쿼리 성능이 떨어지는 경우가 발생합니다. 이럴 때는 성능 향상을 위해 의도적으로 데이터 중복을 허용하는 **반정규화(Denormalization)**를 고려하기도 하지만, 이는 반드시 데이터 불일치 위험을 감수하고 철저히 관리해야 할 고급 기법입니다.
