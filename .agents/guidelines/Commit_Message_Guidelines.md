# 커밋 메시지 작성 가이드라인 (Commit Message Guidelines)

에이전트와 개발자가 커밋을 수행할 때 일관된 이력을 남기기 위해 아래의 형식을 반드시 준수합니다.

---

## 1. 커밋 메시지 기본 형식
모든 커밋 제목은 아래와 같은 구조를 가집니다.

`[분야] 분류: 내용`

- **예시:** `[API] feat: 게시글 검색 필터링 기능 추가`

---

## 2. 분야 (Field) 구분
작업이 영향을 미치는 범위를 명시합니다.

- **[API]**: 서버 라우트, 엔드포인트 관련 작업
- **[DB]**: 스키마 정의, 마이그레이션, 저장소(store) 로직
- **[REDIS]**: 캐싱 로직 및 Redis 관련 설정
- **[UX]**: 프론트엔드 UI/UX 개선 및 사용자 피드백 관련
- **[DOCS]**: 가이드라인 문서 작성 및 README 수정
- **[CHORE]**: 패키지 설치, 환경 설정, 기타 단순 작업

---

## 3. 분류 (Type) 구분
작업의 성격을 정의합니다. (Conventional Commits 기반)

- **feat**: 새로운 기능 추가
- **fix**: 버그 수정
- **docs**: 문서 수정
- **style**: 코드 포맷팅 (내용 변경 없음)
- **refactor**: 코드 리팩토링
- **test**: 테스트 코드 추가/수정
- **chore**: 빌드 업무, 기타 사소한 작업

---

## 4. 작성 원칙
1. **명령조 사용:** "Update logic" (O), "Updated logic" (X)
2. **마침표 지양:** 제목 끝에 마침표(.)를 찍지 않습니다.
3. **50자 제한:** 제목은 가급적 50자 이내로 간결하게 정보를 전달합니다.
4. **본문 활용:** 제목만으로 부족한 경우, 한 줄 띄운 뒤 '무엇을', '왜' 변경했는지 상세히 기록합니다.
5. **Changelog 필수 업데이트:** 기능 추가, 버그 수정, 또는 주요 설정 변경 시 반드시 `CHANGELOG.md`를 업데이트하고 해당 내용도 커밋에 포함하거나 바로 이어서 커밋합니다. 관련 상세 지침은 [Changelog_Guidelines.md](./Changelog_Guidelines.md)를 참조하세요.

---

## 5. 종합 예시
- `[API] feat: 게시글 목록 조회 페이징 처리`
- `[DB] fix: MySQL 연결 시 한글 인코딩 깨짐 해결`
- `[DOCS] docs: Redis 설계 원칙 문서 추가`
- `[CHORE] chore: ioredis 패키지 설치`
